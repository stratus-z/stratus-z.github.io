1. 思考题

Thinking 6.1

如果父子进程不关闭自己不会使用的管道端，管道另一端是否关闭的判断会失效，并可能造成阻塞。

Thinking 6.2

若希望父进程作为读者，只需要交换test_pipe.c中父子进程对管道两端的使用方式：

核心原则是读者只保留 field[0]，写者只保留 field[1]。因此父进程读时应关闭自己的写端 field[1]，子进程写时应关闭自己的读端 field[0]。

Thinking 6.3

PIPE_SIZE 过小会让管道很容易变满或变空。本实验中 user/lib/pipe.c 的 pipe_write 在 p_wpos - p_rpos == PIPE_SIZE 时会循环 syscall_yield()，pipe_read 在 p_rpos == p_wpos 时也会在未关闭时让出 CPU。因此缓冲区越小，读写双方越频繁进入“等待对方”的状态，系统调用和进程切换次数增加，吞吐量下降。

较大的缓冲区可以吸收短时间的读写速率差，减少阻塞和切换，提高连续传输时的吞吐。但缓冲区不是越大越好，过大的缓冲区会浪费内存，也可能增加缓存压力；当缓冲区已经足够覆盖典型突发流量后，继续增大带来的收益会变小。

Thinking 6.4

pageref(rfd) + pageref(wfd) = pageref(pipe) 不是所有时刻都严格成立。它成立的前提是管道处于稳定映射状态：每个打开的读端/写端 fd 都对应一份到同一个 pipe 数据页的映射。

在创建、关闭或复制 fd 的中间状态中，fd 页和 pipe 数据页的映射不是由一个用户态函数一次性完成的，而是通过多次系统调用分别修改。例如关闭管道端时，pipe_close 需要分别 unmap fd 页和 pipe 页；如果两次 unmap 之间发生进程切换，引用计数就可能短暂不同步。dup 中复制数据页和复制 fd 页也有类似问题。

这种不成立可能造成误判。若先解除 pipe 页映射、后解除 fd 页映射，就可能在中间状态出现 pageref(fd) == pageref(pipe)，读者据此误以为写端已经全部关闭。当前代码通过调整 pipe_close 的 unmap 顺序，以及在 _pipe_is_closed 中用 env->env_runs 保证两次 pageref 读取期间没有发生进程切换，降低了这种误判风险。

Thinking 6.5

dup 的竞争问题与 close 类似，根源都是 fd 页和 pipe 数据页的引用计数不是同步修改的。user/testpiperace.c 就构造了父进程反复 dup(p[0], 10)、子进程反复 pipe_is_closed(p[0]) 的场景。

假设子进程已经关闭 p[1]，父进程仍持有 p[0] 和 p[1]，并把 p[0] 复制到 fd 10。稳定状态下，读端 fd 页引用数为 3，pipe 页引用数为 4，因为 pipe 页还包含父进程的写端引用，所以子进程不应认为写端关闭。

如果 dup 先 map 新 fd 页，再 map pipe 数据页，那么在两步之间可能出现 fd 引用数已经增加，而 pipe 引用数尚未增加的状态。例如读端 fd 引用数临时变为 3，pipe 引用数仍为 3，子进程用 _pipe_is_closed 判断时就会误以为另一端关闭。当前 user/lib/fd.c 中 dup 先映射 fd2data(oldfd) 对应的数据页，再映射 oldfd 自身，正是为了避免这个中间状态。

Thinking 6.6

在 MOS 中，普通系统调用可以看作相对于用户进程调度是原子的。用户态通过 syscall_* 包装函数进入内核后，由 kern/genex.S 的 handle_sys 调用 kern/syscall_all.c 中的 do_syscall，再分发到具体 sys_* 函数。系统调用执行期间处理器处于异常处理流程中，在返回用户态前不会像普通用户代码一样被时间片切走，所以 sys_mem_map、sys_mem_unmap 这种单个系统调用对页引用计数的修改可以视为不可被其他用户进程插入。

但这不意味着一串系统调用组成的用户态函数也是原子的。比如 pipe_close 包含两次 syscall_mem_unmap，dup 包含多次 syscall_mem_map，它们之间仍可能被调度。因此实验中需要控制这些系统调用的顺序。

严格来说，如果把“原子”理解为“调用期间绝不发生进程切换并且一定回到原进程继续执行”，则 sys_yield、sys_ipc_recv 和对当前进程调用 sys_set_env_status 是反例。这些函数会显式调用 schedule(1)，主动切换到其他进程。不过它们在切换前对内核状态的修改仍是在内核态连续完成的。

Thinking 6.7

可以解决指导书中描述的 close 竞争问题。关键是保证在“另一端实际上还没关闭”的任何中间状态下，都不要出现 pageref(fd) == pageref(pipe)。当前 user/lib/pipe.c 的 pipe_close 顺序为：

即先解除 fd 页，再解除 pipe 数据页。这样在两次 unmap 的间隙，fd 引用数先减少，而 pipe 引用数还没有减少，因此仍有 pageref(pipe) > pageref(fd)。即使此时发生进程切换，另一进程也不会把“对端仍打开”的状态误判为关闭。

dup 也会出现类似问题。若复制的 fd 指向管道，而 dup 先映射 fd 页、后映射 pipe 数据页，那么中间状态可能让 fd 引用数先增加，导致 pageref(fd) == pageref(pipe) 的假象。当前 user/lib/fd.c 的做法是先复制 fd2data(oldfd) 范围内的数据页，再复制 oldfd 自身。这个顺序与 close 的修正方向相反，但目标一致：在对端未关闭时保持 pipe 页引用数严格大于用于判断的本端 fd 页引用数。

Thinking 6.8

Lab5 中打开文件的过程如下。用户态 open 先调用 fd_alloc 找到空闲 fd 页；随后 fsipc_open 把路径和打开模式放入请求页，通过 IPC 发给文件系统服务进程。服务端 serve_open 调用 open_alloc 分配打开文件表项，根据 O_CREAT 可能调用 file_create，再通过 file_open、walk_path、dir_lookup 找到文件控制块；若有 O_TRUNC，则调用 file_set_size 截断文件。最后服务端填充 Filefd，把 fd 页以 PTE_LIBRARY 映射回用户进程。用户态 open 再根据文件大小循环调用 fsipc_map，由服务端 serve_map 和 file_get_block 把文件内容页映射到用户地址空间。

读取并加载 ELF 时，Lab3 的 load_icode 假设 ELF 已经在内核内存中，先用 elf_from 校验 ELF 头，再遍历 program header，对 PT_LOAD 段调用 elf_load_seg，并用 load_icode_mapper 给目标进程分配页面、复制段内容、建立映射。Lab6 的 spawn 做的是用户态版本：它先 open 可执行文件，readn 读 ELF 头，syscall_exofork 创建子进程，init_stack 准备参数栈，然后对每个 PT_LOAD 段用 read_map 找到文件映射中的段内容，并用 elf_load_seg 加 spawn_mapper 把段装入子进程地址空间，最后设置子进程 EPC、SP，继承 PTE_LIBRARY 页并设为 runnable。

bss 段的正确加载依赖 p_memsz 大于 p_filesz 时的补零逻辑。lib/elfloader.c 中 elf_load_seg 先根据 p_filesz 把文件中实际存在的 text/data 内容复制进内存；随后在 while (i < sgsize) 中继续为 p_memsz 剩余部分调用 mapper，且传入 src == NULL。在 Lab3 的 load_icode_mapper 中，新页由 page_alloc 分配，而 page_alloc 会 memset 清零；在 Lab6 的 spawn_mapper 中，syscall_mem_alloc 最终也分配零页，只有 src != NULL 时才复制文件内容。因此 ELF 文件中不占空间的 bss 在虚拟内存中获得了实际页面，并保持初值 0。

Thinking 6.10

MOS 中使用的 shell 命令是外部命令。user/sh.c 的 runcmd 不区分 ls、cat、echo 等命令类型，而是统一调用：

因此命令对应的是文件系统中的 *.b 可执行文件，例如 ls.b、cat.b，由 spawn 加载到新进程中执行。shell 自身只内置了解析空白、重定向和管道的能力。

Linux 的 cd 必须是内置命令，因为当前工作目录是进程属性。若 cd 作为外部程序运行，它只能改变子进程自己的当前目录；子进程退出后，父 shell 的当前目录不会改变，用户后续命令仍然在原目录下执行。因此 cd 必须直接修改 shell 进程本身的状态。

可以观察到 4 次与该命令相关的进程销毁：

  1. ls.b 执行结束并退出。
  2. cat.b 从管道读到 EOF，写入 motd 后退出。
  3. 管道右侧的子 shell 等待 cat.b 后退出。
  4. 管道左侧的命令执行子 shell 等待 ls.b 和右侧子 shell 后退出。

最外层交互式 shell 不退出，它只是在 main 循环中等待执行命令的子 shell 结束，然后继续读取下一条命令。

2. 难点分析

2.1 管道关闭判断中的引用计数竞争

管道关闭判断的目标是：对读端来说，写端关闭当且仅当 pageref(read_fd) == pageref(pipe)；对写端来说同理。但这个判断依赖 fd 页和 pipe 页引用计数同步变化，而 close、dup 都由多次系统调用组成，不能把整个函数当作原子操作。

2.3 spawn 与 bss 补零

spawn 的关键不只是“读 ELF 文件”，还包括在用户态为另一个进程建立地址空间。spawn_mapper 只能通过系统调用操作子进程内存，因此它先在子进程分配目标页，再把该页临时映射到本进程 UTEMP，然后复制文件内容。

bss 的处理是 ELF 加载中最容易忽略的部分。p_filesz 表示文件中实际占用的字节数，p_memsz 表示加载到内存后应占的字节数。elf_load_seg 对 p_filesz 之后到 p_memsz 的部分继续分配页面但不复制文件内容，而新页初值为 0，所以未初始化全局变量在内存中自然为 0。

3. 实验体会

Lab6 把前面几次实验的机制串在了一起：文件系统提供可执行文件，spawn 负责加载 ELF，PTE_LIBRARY 让文件描述符、管道和控制台能跨 fork/spawn 共享，shell 再把这些机制组合成管道和重定向。

本次实验最有启发的是管道关闭判断。最开始容易把 close 或 dup 看作一个整体动作，但源码里它们其实由多个系统调用拼出来；只要中间发生调度，引用计数就会出现短暂不一致。通过调整 map/unmap 顺序来维护“不误判”的不变量，比简单地追求某个等式时时成立更实际。

另一个体会是，shell 的实现虽然代码量不大，但它强依赖清晰的文件描述符模型。0 和 1 只是约定好的 fd 编号，重定向和管道的本质都是把正确的对象 dup 到这些编号上，然后让后续 spawn 出来的程序继承它们。
